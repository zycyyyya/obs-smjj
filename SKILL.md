# Obsidian私募基金尽调知识库 - LLM Wiki v3.5

> Karpathy框架：单个源可能触及10-15个Wiki页面。全量摄入，一次读完永不再读。

## 核心原则

### 零重读原则（最高优先级）
> **raw/ 中的每个PDF被Ingest后即"消费完毕"，后续所有对话通过 wiki/sources/ 摘要页获取信息，绝不重复读取PDF。**

### 全量摄入
> **Ingest一个管理人 = 列出raw目录下所有PDF → 逐一读取 → 生成sources摘要 → 综合写入管理人档案 → 交叉引用，单次产出 10+ wiki 页面。**

### 严禁偷懒（最高优先级）
> **不允许说"由于token限制"、"加速处理"、"简化处理"、"由于时间和token限制"等任何借口。必须严格按10步流程，一家一家来，不跳过任何步骤。**

### 🛡️ 防偷懒铁律（自我监督）

> 每次执行 Ingest 时，Agent 必须在开始前、中、后执行以下自检。这不再是建议，是硬性要求。

**开始前（Step 0）— 公开承诺**：
```
Agent 必须在第一轮输出中打印：
████████████████████████████████
  开始处理: {管理人}
  文件数: N 个PDF
  预期步骤: Step 0 → Step 9（共10步）
  预计产出: ≥ N+4 个wiki页面
████████████████████████████████
```
> 没有这个输出 = 流程未启动。用户看到这个框才算开始。

**执行中 — 步骤计数器**：
```
每个 Step 执行完必须打印进度：
  ✅ Step X/9 — {步骤名} — 产出: {具体文件名和大小}
  ⏳ 下一步: Step X+1 — {下一步名称}
```
> 进度条必须连续出现。如果用户发现某步跳过了，整个流程作废重来。

**执行中 — 禁止行为清单**：

| 禁止行为 | 原因 |
|---------|------|
| ⛔ 用子代理/并行处理代替主agent串行执行 | 子代理会跳过步骤、简化摘要、绕过检查点 |
| ⛔ 用"批量脚本"一次性生成所有页面 | 脚本无法保证每个摘要的质量和完整性 |
| ⛔ 用"后续再补"跳过当前步骤 | 没有"后续"，每步必须当场完成 |
| ⛔ 用"内容较少"省略某个摘要的模块 | 哪怕只有1页PDF也要写完整摘要模板 |
| ⛔ 用"与已有信息一致"跳过档案更新 | 每次Ingest必须更新档案，无可跳过 |
| ⛔ 空检 — 检查点敷衍了事 | 检查点必须展示真实内容，不能只说"✅ OK" |

**结束后 — 门槛自检**：
```
Step 9 完成后必须打印产出报告，逐项打勾：

📊 产出报告 — {管理人}
  Sources 摘要: N/N 个PDF ✅/❌
  管理人档案: {文件名} ({大小}KB) ✅/❌
  人员页面: {N} 个 ✅/❌
  策略更新: {N} 个 ✅/❌
  产品页面: {N} 个 ✅/❌
  总产出: {N} 页（门槛: ≥10） ✅/❌

  🔒 自检: [PASS / FAIL]
  FAIL 时必须列出补齐清单并立即执行，不能留到下次
```
> 没有这个产出报告 = 流程未完成。自检 FAIL 但不补齐 = 同样未完成。

## PDF 读取工具

### 首选 pdfplumber（稳定可靠）

**多文件循环读取**（用 `tmp_extract/{管理人}.json` 缓存避免重复提取）：
```bash
/c/Users/Administrator/AppData/Local/Programs/Python/Python313/python.exe -c "
import pdfplumber, os, json
m = '{管理人}'
raw_dir = fr'G:\Obsidian\橙子的资料库\raw\{m}'
cache = {}
for fn in os.listdir(raw_dir):
    if fn.endswith('.pdf'):
        try:
            with pdfplumber.open(os.path.join(raw_dir, fn)) as pdf:
                text = '\n'.join([p.extract_text() or '' for p in pdf.pages])
                cache[fn] = text
                print(f'{fn}: {len(text)} chars')
        except Exception as e:
            cache[fn] = f'[FAIL: {e}]'
            print(f'{fn}: FAIL')
out = fr'G:\Obsidian\橙子的资料库\tmp_extract\{m}.json'
with open(out, 'w', encoding='utf-8') as f:
    json.dump(cache, f, ensure_ascii=False, indent=2)
print('Done')
"
```

**备选方案**：加密PDF/乱码时，跳过并在sources摘要中标注"提取失败，待人工录入"。

### 完整流程（共10步，每步必须执行，不可跳过）


## 3️⃣ Lint（健康检查）

**触发**：用户说"健康检查"、"Lint"或每月定期执行

### 执行流程

```
Step L1  扫描 wiki/index.md 获取所有已标记 v3_ingested 的管理人清单
Step L2  逐管理人检查以下7个维度
Step L3  生成 Lint 报告
Step L4 ⏸️ 展示报告，询问用户"需要修复哪些问题？"
          （不自动修复，等用户选择后再执行）
```

### 7个检查维度

| # | 维度 | 检查方法 | 通过标准 |
|---|------|---------|---------|
| L1 | Sources完整性 | 比对 raw/{管理人}/ 文件列表 vs wiki/sources/ 摘要数 | 每个PDF有对应摘要 |
| L2 | v3_ingested一致性 | 检查档案8模块完整性 + 人员页面 + 产品页面 | 全部存在且非空 |
| L3 | 孤岛页面 | 扫描所有 wiki/*.md 的入站链接 | 每个页面至少有1个入站链接 |
| L4 | 死链 | 扫描所有 `[[链接]]` 是否指向存在的文件 | 无断链 |
| L5 | 矛盾信息 | 对比 sources摘要 vs 档案页关键字段（规模/成立时间） | 无明显矛盾 |
| L6 | 时效性 | 检查 log.md 最后更新时间 | ≤6个月 |
| L7 | 产出门槛 | 统计每家v3_ingested管理人的总产出页面数 | ≥10页 |

## 批量补齐注意事项（v3.3 经验总结）

> 基于2026-05-25~27日实战总结

### 关键教训

1. **必须先读完skill再开干** — 不要跳过skill直接写脚本
2. **严禁说"由于token限制"或"加速处理"** — 这是偷懒的借口，用户明确禁止
3. **严禁简化sources摘要** — 每个摘要必须完整，不能写"待补充"
4. **文件命名以档案中`[[链接]]`格式为准** — 不是公司全称，而是档案中实际使用的简称
5. **提取链接时清理转义字符** — Markdown表格中的`\|`需用`replace('\\', '')`清理
6. **中文引号会导致Python SyntaxError** — 在Python字符串中把`"`替换为`「」`
7. **用`[[文件名|显示文本]]`格式时取`|`前部分** — `link.split('|')[0]`
8. **不能用`is_likely_personnel()`模糊判断链接类型** — 策略名/来源文件/Q&A会被误判，应精确解析核心团队表格
9. **标准模板不可简化** — 人员页面必须有6个标准模块，产品页面必须有完整业绩数据
10. **一家一家来，不批量处理** — 每次只处理一家管理人，确保质量

### 批量补齐标准流程

```
1. 读取管理人档案 → 提取核心团队表格中的 [[人员链接]] 和产品表格
2. 检查每个链接是否已有对应页面（精确匹配，注意简称vs全称）
3. 缺失的页面按标准模板创建（不可用简化模板）
4. 已有但内容简化的页面 → 按标准模板扩充重建
5. 全部完成后验证：每家的档案中所有 [[链接]] 都有对应页面
6. Lint检查：人员链接匹配、产品链接匹配
```

### 常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| 用全称创建页面 | 以档案`[[链接]]`格式为准 |
| `\|`转义符污染链接名 | `replace('\\', '')` |
| 中文引号SyntaxError | 替换为`「」` |
| 模糊判断链接类型 | 精确解析表格结构 |
| 简化模板 | 必须含6个标准模块 |
| 批量处理多家 | 一家一家来，确保质量 |
| 跳过Step 7-9 | 每步必须执行 |
